# 백엔드 모듈 코드 리뷰 파이프라인 가이드

이 문서는 현재 v1 경로에 정의된 스킬들을 활용하여, 모듈(기능 단위)별 코드 리뷰를 효과적으로 진행하기 위한 단계별 전략과 프롬프트 템플릿을 제공합니다. 

거시적(Macro) 관점에서 시작하여 미시적(Micro) 관점, 그리고 데이터 및 테스트 검증으로 범위를 좁혀가며 스킬을 연결해 사용하는 파이프라인(Pipeline) 방식을 권장합니다.

---

## 단계별 스킬 활용 전략 (Code Review Pipeline)

### 1단계: 구조 및 아키텍처 점검 (Macro)
가장 먼저 해당 모듈이 전체 시스템 내에서 올바른 구조를 가지고 있는지, 외부 컴포넌트와의 경계를 제대로 설정했는지 확인합니다.

*   **적용 대상:** 모듈의 패키지 구조, 레이어 계층, 외부로 노출하는 인터페이스 구조 등
*   **관련 스킬:** 
    *   `/architecture-review`: 패키지 분리 여부, 계층(Layer) 간의 의존성 검토
    *   `/api-contract-review`: REST API 컨트롤러 설계, HTTP 메서드와 상태 코드의 원칙 준수 여부 점검
*   **활용 예시 프롬프트:**
    > "새로 개발한 `[모듈 이름]` 모듈의 전체적인 구조를 리뷰해줘. (적용 스킬: `/architecture-review`)"
    > "이 모듈에서 제공하는 REST API 컨트롤러의 설계가 적절한지 점검해줘. (적용 스킬: `/api-contract-review`)"

### 2단계: 핵심 비즈니스 로직 및 객체지향 설계 점검 (Meso)
전체 구조가 잡혔다면, 모듈 내부의 핵심 도메인 로직과 클래스 설계가 잘 짜여 있는지 검토합니다.

*   **적용 대상:** 비즈니스 로직을 담당하는 Service, Domain 클래스 등
*   **관련 스킬:**
    *   `/solid-principles`: 단일 책임 원칙(SRP), 개방-폐쇄 원칙(OCP) 등 객체지향 설계 준수 여부
    *   `/design-patterns`: 팩토리(Factory), 전략(Strategy) 등 적절한 디자인 패턴 적용 가능성 모색
    *   `/clean-code`: 거대한 클래스, 중복 코드(DRY), 복잡한 로직 정리 및 명명 규칙 점검
*   **활용 예시 프롬프트:**
    > "이 모듈의 핵심 서비스 클래스들을 리뷰해줘. (적용 스킬: `/solid-principles`, `/design-patterns`)"
    > "로직 중에 너무 복잡하거나 역할이 비대한 부분이 있다면 클린 코드 관점에서 리팩토링 방향을 제안해줘. (적용 스킬: `/clean-code`)"

### 3단계: 성능, 안전성, 데이터베이스 연동 점검 (Micro/Data)
세부적인 구현 단계에서의 문제점, 특히 성능 저하나 보안 취약점, 데이터베이스 이슈를 깊게 파고듭니다.

*   **적용 대상:** JPA 엔티티(Entity), 레포지토리(Repository), 복잡한 연산 보조 로직, 외부 통신 어댑터 등
*   **관련 스킬:**
    *   `/jpa-patterns`: N+1 문제, 지연 로딩(Lazy Loading) 이슈, 트랜잭션 관리 점검
    *   `/performance-smell-detection`: 불필요한 객체 생성(Boxing), 비효율적인 Stream 처리, 무거운 정규식 반복 통제
    *   `/concurrency-review`: 데드락, 스레드 안전성(Thread safety), 경쟁 상태 점검
    *   `/security-audit`: 사용자 입력값 검증, 주요 공격(SQL Injection 등) 방어, OWASP 취약점 체크
*   **활용 예시 프롬프트:**
    > "이 모듈의 JPA 계층 코드를 리뷰하고 N+1 문제 같은 성능 저하 요인이 없는지 확인해줘. (적용 스킬: `/jpa-patterns`)"
    > "스트림 처리나 반복 연산에서 성능 낭비를 일으키는 부분이 없는지 꼼꼼히 리뷰해줘. (적용 스킬: `/performance-smell-detection`)"
    > "동시성 이슈나 OWASP 보안 취약점이 발생할 여지가 있는지 점검해줘. (적용 스킬: `/concurrency-review`, `/security-audit`)"

### 4단계: 테스트 및 스타일 최종 검토
기능 보장을 담당하는 테스트 코드의 품질을 리뷰하고, 코드 스타일 및 오류 처리를 마지막으로 종합 검사합니다.

*   **적용 대상:** 단위 테스트 및 통합 테스트 코드 모음, 전반적인 소스 코드 파일들
*   **관련 스킬:**
    *   `/test-quality`: JUnit 5, AssertJ를 활용한 가독성 좋고 유지보수 용이한 패턴, 커버리지 검토
    *   `/java-code-review`: 누락된 예외 처리 확인, 포괄적인 Java 코드 스타일 및 모범 사례 점검
*   **활용 예시 프롬프트:**
    > "이 모듈에 작성된 테스트들의 가독성이나 맹점은 없는지 평가해줘. (적용 스킬: `/test-quality`)"
    > "전체 코드 중에서 Java 생태계의 일반적인 Best Practice에 어긋나거나 에러 핸들링이 미흡한 곳이 없는지 마지막으로 검토해줘. (적용 스킬: `/java-code-review`)"

## 💡 실전 활용 팁: 통합 코드 리뷰 프롬프트 템플릿

실제로 AI(Claude 등)에게 모듈 리뷰를 맡길 때는, 여러 스킬을 연속적으로 지시하고 중요도(Severity) 기반으로 결과를 출력하도록 아래와 같은 템플릿을 활용하는 것이 가장 뛰어난 결과를 만들어냅니다.

```text
현재 `[모듈 이름]` 모듈에 대한 전반적인 코드 리뷰를 진행하려고 해. 
다음과 같은 관점에서 순차적으로 코드를 분석하고 피드백을 작성해줘.

1. 아키텍처 및 API 설계 검토 (`/architecture-review`, `/api-contract-review`)
2. 핵심 도메인의 객체지향 설계와 가독성 검토 (`/solid-principles`, `/clean-code`)
3. 영속성 계층 구현 및 실행 성능 검토 (`/jpa-patterns`, `/performance-smell-detection`)
4. 잠재적인 동시성 및 보안 취약점 포인트 검토 (`/concurrency-review`, `/security-audit`)
5. 기능 보장을 위한 테스트 코드 품질 평가 (`/test-quality`)

모든 문제를 나열만 하지 말고, 수정의 시급성을 나타내는 중요도(Critical, Major, Minor) 순서대로 리팩토링 제안과 함께 요약 정리해줘.
```
